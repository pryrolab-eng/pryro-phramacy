# Sales Module

## Purpose

The Sales module provides a read-only reporting and analytics dashboard for completed pharmacy transactions. It gives pharmacy staff a consolidated view of sales performance across multiple time horizons (today, this week, this month, all time), a searchable and filterable transaction history, and visual analytics covering payment method distribution, top product categories, weekly trends, hourly patterns, and month-over-month comparisons.

The Sales module is distinct from the Point of Sale (POS) module. POS (`/pos`) is where transactions are created in real time. Sales (`/sales`) is where completed transactions are reviewed and analysed after the fact.

### API Routes (`src/app/api/sales/`)

| File | Route | Method | Description |
|---|---|---|---|
| `route.ts` | `/api/sales` | `GET` | Returns the 20 most recent sales for the authenticated user's pharmacy, plus aggregated stats (today total, week total, month total, transaction count). Requires an authenticated session; returns empty data for unauthenticated requests rather than a 401. |
| `route.ts` | `/api/sales` | `POST` | Creates a new sale record in `sales`, inserts line items into `sale_items`, and decrements `inventory.quantity_in_stock` for each item. Used by the POS module to persist completed transactions. |
| `analytics/route.ts` | `/api/sales/analytics` | `GET` | Computes and returns six analytics datasets: weekly sales by day, payment method breakdown (last 30 days), hourly sales for today, monthly comparison (current vs. previous month by week), top product categories (last 30 days), and customer distribution (walk-in / regular / insurance). |

## Database Tables

### `sales`

Stores one record per completed POS transaction.

| Column | Type | Description |
|---|---|---|
| `id` | `uuid` | Primary key |
| `pharmacy_id` | `uuid` | Foreign key → `pharmacies.id` (CASCADE DELETE). Enforces tenant isolation. |
| `cashier_id` | `uuid` | Foreign key → `auth.users.id`. The staff member who processed the sale. |
| `customer_name` | `text` | Customer display name. Defaults to `'Walk-in Customer'` when no customer is selected. |
| `customer_phone` | `text` | Customer phone number (nullable). |
| `insurance_provider_id` | `uuid` | Foreign key → `insurance_providers.id` (nullable). Set when insurance covers part of the sale. |
| `subtotal` | `decimal(10,2)` | Pre-insurance total of all line items. |
| `insurance_amount` | `decimal(10,2)` | Amount covered by the insurance provider. |
| `customer_amount` | `decimal(10,2)` | Amount paid directly by the customer (`subtotal - insurance_amount`). |
| `total_amount` | `decimal(10,2)` | Final transaction total (equals `subtotal`). |
| `payment_method` | `payment_method` enum | One of: `cash`, `card`, `mobile_money`, `insurance`, `mixed`. |
| `status` | `sale_status` enum | One of: `completed`, `pending`, `cancelled`, `refunded`. Default: `completed`. |
| `rra_invoice_number` | `text` | Rwanda Revenue Authority invoice number (nullable; for tax compliance). |
| `receipt_number` | `text` | Human-readable receipt identifier (e.g., `RCP-2024-001`). Generated by the POS as `RCP-<timestamp>`. |
| `notes` | `text` | Optional free-text notes on the transaction. |
| `created_at` | `timestamptz` | Transaction timestamp. |
| `updated_at` | `timestamptz` | Last update timestamp (auto-maintained by trigger). |

**Indexes:** `idx_sales_pharmacy_id` on `pharmacy_id`; `idx_sales_created_at` on `created_at`.

**Realtime:** The `sales` table is added to the `supabase_realtime` publication, so new sales are broadcast to subscribed clients immediately.

### `sale_items`

Stores one record per line item within a sale.

| Column | Type | Description |
|---|---|---|
| `id` | `uuid` | Primary key |
| `sale_id` | `uuid` | Foreign key → `sales.id` (CASCADE DELETE). |
| `inventory_id` | `uuid` | Foreign key → `inventory.id`. The specific inventory batch sold. |
| `medication_name` | `text` | Snapshot of the product name at time of sale (denormalised for receipt stability). |
| `quantity` | `integer` | Number of units sold. |
| `unit_price` | `decimal(10,2)` | Price per unit at time of sale. |
| `total_price` | `decimal(10,2)` | `quantity × unit_price`. |
| `batch_number` | `text` | Inventory batch number (nullable; snapshot for traceability). |
| `expiry_date` | `date` | Expiry date of the batch sold (nullable; snapshot for traceability). |
| `created_at` | `timestamptz` | Record creation timestamp. |

**Index:** `idx_sale_items_sale_id` on `sale_id`.

**Realtime:** The `sale_items` table is added to the `supabase_realtime` publication.

**Trigger:** `handle_sale_stock_update_trigger` fires `AFTER INSERT` on `sale_items`. It decrements `inventory.quantity_in_stock` by `NEW.quantity` and inserts a `stock_movements` record with `movement_type = 'out'` and `reference_type = 'sale'`.

### Enum Types

| Type | Values |
|---|---|
| `payment_method` | `cash`, `card`, `mobile_money`, `insurance`, `mixed` |
| `sale_status` | `completed`, `pending`, `cancelled`, `refunded` |

## Row-Level Security

All RLS policies on `sales` and `sale_items` use the helper function `get_user_pharmacy_ids()`, which returns the set of pharmacy IDs the authenticated user belongs to. This enforces strict tenant isolation — a user can only see and modify sales belonging to their own pharmacy.

| Table | Policy | Operation | Condition |
|---|---|---|---|
| `sales` | Pharmacy staff can view sales | `SELECT` | `pharmacy_id = ANY(get_user_pharmacy_ids())` |
| `sales` | Pharmacy staff can create sales | `INSERT` | `pharmacy_id = ANY(get_user_pharmacy_ids())` |
| `sales` | Pharmacy staff can update sales | `UPDATE` | `pharmacy_id = ANY(get_user_pharmacy_ids())` |
| `sale_items` | Pharmacy staff can view sale items | `SELECT` | `sale_id IN (SELECT id FROM sales WHERE pharmacy_id = ANY(...))` |
| `sale_items` | Pharmacy staff can manage sale items | `ALL` | `sale_id IN (SELECT id FROM sales WHERE pharmacy_id = ANY(...))` |

There is no `DELETE` policy on `sales`, meaning sales records cannot be deleted through the application layer.

## Features

### KPI Summary Cards

The top of the page displays four metric cards populated from `/api/sales`:

| Card | Metric | Source |
|---|---|---|
| Today's Sales | Sum of `total_amount` for sales created today | `sales` WHERE `created_at >= today` |
| This Week | Sum of `total_amount` for the last 7 days | `sales` WHERE `created_at >= 7 days ago` |
| This Month | Sum of `total_amount` for the last 30 days | `sales` WHERE `created_at >= 30 days ago` |
| Total Transactions | Count of the 20 most recent sales returned | `sales` LIMIT 20 |

All monetary values are displayed in RWF (Rwandan Franc). The percentage-change labels ("+15% from yesterday", etc.) are **hardcoded strings** — they are not computed from real data.

### Sales History (Transactions Tab)

A paginated table showing the 20 most recent sales with columns: Customer, Items, Amount, Payment Method, Date, Status.

**Client-side filtering** is applied after data is fetched:

- **Search:** Filters by `customer` name or `paymentMethod` (case-insensitive substring match).
- **Period selector:** Filters by `today`, `this week` (last 7 days), `this month` (last 30 days), or `all time`. Filtering is done in the browser against the already-fetched 20-record window.

> **Limitation:** Because the API always returns only the 20 most recent records, the period filter and search operate on a capped dataset. Selecting "This Month" will only show matching records from the most recent 20, not all sales in the month.

### Receipt Reprint

The Sales page does **not** implement receipt reprinting. The `Export` button in the page header renders a `<Download>` icon but has no `onClick` handler — it is a non-functional UI placeholder.

Receipt printing is implemented in the POS module (`src/app/(dashboard)/pos/page.tsx`) via the `printInvoice()` function, which opens a browser print dialog with an HTML receipt template at the time of sale. There is no mechanism to reprint a historical receipt from the Sales page.

### Export to CSV / PDF

The Sales page does **not** implement CSV or PDF export. The `Export` button is present in the UI but is not wired to any export logic.

For comparison, the Inventory module (`src/app/(dashboard)/inventory/page.tsx`) implements a working Excel export using the `xlsx` library, and the Reports module (`src/app/(dashboard)/reports/page.tsx`) implements a PDF export via `window.print()`. Neither of these patterns has been applied to the Sales page.

### Overview Tab — Charts

The Overview tab renders four visualisation components:

| Component | Chart Type | Data Source | Fallback |
|---|---|---|---|
| Weekly Sales Trend | Line chart (Recharts) | `/api/sales/analytics` → `weeklySales` | Hardcoded 7-day sample data |
| Today's Hourly Sales | Line chart (Recharts) | `/api/sales/analytics` → `hourlySales` | Hardcoded 8-hour sample data |
| Payment Methods | Progress bars | `/api/sales/analytics` → `paymentBreakdown` | Empty (no fallback rendered) |
| Top Categories | Progress bars | `/api/sales/analytics` → `topCategories` | Hardcoded 4-category sample data |
| Recent Sales | Scrollable list | `/api/sales` → first 5 records | Empty list |

### Analytics Tab — Charts

| Component | Chart Type | Data Source | Fallback |
|---|---|---|---|
| Sales Performance | Grouped bar chart (Recharts) | `/api/sales/analytics` → `monthlyComparison` | Hardcoded 4-week sample data |
| Customer Distribution | Donut pie chart (Recharts) | `/api/sales/analytics` → `customerDistribution` | Hardcoded 3-segment sample data |

Customer distribution is classified as:
- **Insurance** — sales with a non-null `insurance_provider_id`
- **Regular** — sales with a named customer (not `'Walk-in Customer'`)
- **Walk-in** — all other sales

## Data Flow

```
Browser (SalesPage)
    │
    ├─ useEffect → fetch('/api/sales')
    │       │
    │       └─ GET /api/sales
    │               │
    │               ├─ supabase.auth.getUser() → verify session
    │               ├─ pharmacy_users → resolve pharmacy_id
    │               ├─ sales.select('*').eq('pharmacy_id', ...).limit(20)
    │               ├─ 3× aggregation queries (today / week / month totals)
    │               └─ { sales: Sale[], stats: Stats }
    │
    ├─ useEffect → fetch('/api/sales/analytics')
    │       │
    │       └─ GET /api/sales/analytics
    │               │
    │               ├─ supabase.auth.getUser() → verify session
    │               ├─ pharmacy_users → resolve pharmacy_id
    │               ├─ sales (last 7 days) → weeklySales
    │               ├─ sales (last 30 days) → paymentBreakdown
    │               ├─ sales (today) → hourlySales
    │               ├─ sales (current + previous month) → monthlyComparison
    │               ├─ sale_items JOIN sales JOIN inventory JOIN medications → topCategories
    │               ├─ sales (last 30 days) → customerDistribution
    │               └─ { weeklySales, paymentBreakdown, hourlySales,
    │                    monthlyComparison, topCategories, customerDistribution }
    │
    └─ Client-side filter (search + period) → filteredSales → render table
```

## Known Limitations

### 1. Transaction list is capped at 20 records

`GET /api/sales` always returns `LIMIT 20`. The client-side period filter and search operate on this 20-record window. A pharmacy with more than 20 sales today will see an incomplete "Today" view. There is no pagination, infinite scroll, or server-side filtering.

### 2. Export button is non-functional

The `Export` button in the page header renders correctly but has no `onClick` handler. Clicking it does nothing. CSV and PDF export are not implemented for the Sales module.

### 3. Receipt reprint is not available

There is no way to reprint a historical receipt from the Sales page. Receipt printing only occurs at the moment of sale in the POS module.

### 4. Percentage-change labels are hardcoded

The KPI cards display "+15% from yesterday", "+8% from last week", "+12% from last month", and "+5% from yesterday". These strings are static and do not reflect actual period-over-period changes.

### 5. `item count` is hardcoded in the API

`GET /api/sales` maps each sale to `items: 2` (a hardcoded mock value). The actual item count requires a join to `sale_items`, which is not performed. The Transactions table therefore always shows "2" in the Items column regardless of the real line-item count.

### 6. Analytics API uses a fragile join for top categories

`GET /api/sales/analytics` attempts a nested join: `sale_items → sales → inventory → medications`. This join pattern (`inventory!inner(medications!inner(category))`) may not work correctly with Supabase's PostgREST query syntax and is likely to return empty `topCategories` data in production, causing the chart to fall back to hardcoded sample data.

### 7. `/api/accounting` is a stub

`GET /api/accounting` returns entirely hardcoded data (revenue, expenses, profit, monthly breakdown). It is not connected to the database and is not consumed by any page in the current codebase. It should either be implemented or removed.

### 8. Pharmacist role has no Sales link

The `PharmacistSidebar` does not include a link to `/sales`. Pharmacists who need to review sales history must navigate to the URL directly. This is likely an oversight rather than an intentional access restriction, since RLS permits all pharmacy staff to read sales data.

### 9. No error state in the UI

If `/api/sales` or `/api/sales/analytics` fails, the page silently falls back to hardcoded mock data (defined in the `catch` blocks of both API routes and the page's `fetchSales` function). The user receives no indication that the displayed data is not real.

### 10. `supabase.raw()` call in POST handler is invalid

`POST /api/sales` attempts to decrement inventory stock using `supabase.raw('quantity_in_stock - ?', [item.quantity])`. The `supabase-js` client does not have a `.raw()` method — this is a Knex.js pattern. This call will throw a runtime error, meaning inventory stock is **not decremented** when a sale is created via the API. Stock deduction only works correctly when triggered by the database-level `handle_sale_stock_update_trigger` (which fires on `sale_items` INSERT).
